# Gestión de Credenciales

Laravel MercadoPago usa un sistema de resolución flexible que separa la obtención de la credencial de la configuración misma del SDK. Esto permite soportar configuraciones basadas en el entorno (`.env`), bases de datos, secret managers y ambientes multi-tenant (multi-inquilino).

## Tipos de Credenciales

Mercado Pago utiliza tres tipos de credenciales:

  <Card title="Token de Acceso" icon="key">

  </Card>
  <Card title="Clave Pública" icon="unlock">
    **Opcional** - Usada para inicializar el SDK del lado del cliente (JS)
  </Card>
  <Card title="Webhook Secret" icon="shield-check">
    **Opcional** - Valida firmas de webhook vía HMAC-SHA256
  </Card>

### Access Token (Token de Acceso)

**Propósito**: Autenticación del lado del servidor para todas las operaciones en la API.

**Requerido para**: Todos los servicios excepto WebhookService (cuando no se valida la firma).

**Formato**: Un texto largo con caracteres alfanuméricos (ej. `APP_USR-123456789-abcdef-...`)

<Warning>
  El token de acceso le da control total sobre la cuenta de Mercado Pago asociada. NUNCA se lo debe añadir al control de versiones (git) ni enviarlo o exponerlo en el código de front-end.
</Warning>

### Public Key (Clave Pública)

**Propósito**: Inicializar el SDK del lado del frontend para, por ejemplo, tokenizar tarjetas.

**Requerido para**: Integraciones directas de frontend que usen el SDK.js de Mercado Pago.

**Formato**: Un identificador de clave pública (ej. `APP_USR-abcdef123456-...`)

<Info>
  La clave pública puede exponerse de manera segura en el cliente (navegador). Solamente sirve para funciones como tokenizar elementos con seguridad sin tener acceso de lectura/escritura a la API completa.
</Info>

### Webhook Secret (Secreto del Webhook)

**Propósito**: Realización de validaciones HMAC-SHA256 contra la firma de las notificaciones Webhook entrantes.

**Requerido para**: Validación de la autenticidad segura del webhhook.

**Formato**: Una cadena alfanumérica única obtenida en el panel de Mercado Pago.

<Tip>
  Siempre debés configurar este secreto en tus ambientes de producción para evitar notificaciones o transacciones falsificadas desde el exterior.
</Tip>

## Resolución de Credenciales

### DTO MercadoPagoCredentials

Las credenciales viajan envueltas en un Objeto de Transferencia de Datos (DTO) de sólo lectura:

**Ubicación**: `src/DTO/MercadoPagoCredentials.php`

```php
final readonly class MercadoPagoCredentials
{
    public function __construct(
        public string $accessToken,
        public ?string $publicKey = null,
        public ?string $webhookSecret = null,
    ) {}
}
```

**Beneficios de este enfoque DTO**:

- **Type Safety**: El access token está garantizado como una variable string.
- **Inmutabilidad**: Prohíbe mutar dinámicamente credenciales críticas luego de ser instanciadas.
- **Claridad**: Marca las propiedades accesorias (public y webhook) de modo explícitamente `null`ables.

### CredentialResolverInterface

El resolver o contrato maestro que dictamina que la credencial fue extraída desde un punto seguro:

**Ubicación**: `src/Contracts/CredentialResolverInterface.php`

```php
interface CredentialResolverInterface
{
    public function resolve(): MercadoPagoCredentials;
}
```

### EnvCredentialResolver

La implementación estándar y por defecto que trae el paquete que extrae configuraciones del archivo de de configuración de Laravel vía `.env`.

**Ubicación**: `src/Support/EnvCredentialResolver.php`

```php
final readonly class EnvCredentialResolver implements CredentialResolverInterface
{
    //...
    public function resolve(): MercadoPagoCredentials
    {
        $accessToken = (string) $this->config->get('mercadopago.access_token', '');

        if ($accessToken === '') {
            throw MercadoPagoConfigurationException::missingAccessToken();
        }

        return new MercadoPagoCredentials(
            accessToken: $accessToken,
            publicKey: $this->config->get('mercadopago.public_key') ?: null,
            webhookSecret: $this->config->get('mercadopago.webhook_secret') ?: null,
        );
    }
}
```

## Creando Resolutores Dinámicos (Multi Tenant / DB)

Si utilizas entornos que atienden a muchos comercios en simultáneo, podés anular el `EnvCredentialResolver`.

```php
namespace App\Services;

use Fitodac\LaravelMercadoPago\Contracts\CredentialResolverInterface;
use Fitodac\LaravelMercadoPago\DTO\MercadoPagoCredentials;

class TenantCredentialResolver implements CredentialResolverInterface
{
    public function resolve(): MercadoPagoCredentials
    {
        $tenant = tenant(); // Lógica local de tu SaaS

        return new MercadoPagoCredentials(
            accessToken: $tenant->mp_access_token,
            publicKey: $tenant->mp_public_key,
            webhookSecret: $tenant->mp_webhook_secret,
        );
    }
}
```

Y lo atás al container (preferiblemente de modo Transient / Scoped, y no singleton):

```php
public function register()
{
    $this->app->bind(
        CredentialResolverInterface::class,
        TenantCredentialResolver::class
    );
}
```

<Warning>
  Al usar multi-tenancy, tené en cuenta de que el container o los factories se deben despachar individualmente; de lo contrario podrías llegar a mezclar tokens de clientes diferentes.
</Warning>
