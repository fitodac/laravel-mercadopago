## Bloques del paquete

El paquete se apoya en cuatro capas:

1. `LaravelMercadoPagoServiceProvider`
2. capa HTTP para endpoints demo
3. services que envuelven clientes del SDK de Mercado Pago
4. clases de soporte para credenciales, configuración del SDK y normalización de respuestas

## Responsabilidades del service provider

`LaravelMercadoPagoServiceProvider` hace tres cosas:

- mergea `config/mercadopago.php`
- bindea `CredentialResolverInterface`
- registra singleton services y rutas

Singletons registrados:

- `MercadoPagoClientFactory`
- `SdkHttpClient`
- `PreferenceService`
- `PaymentService`
- `RefundService`
- `CustomerService`
- `CardService`
- `PaymentMethodService`
- `WebhookService`
- `TestUserService`

## Modelo de rutas

El paquete carga rutas desde `routes/api.php` bajo:

```php
Route::prefix((string) config('mercadopago.route_prefix', 'api/mercadopago'))
    ->name('mercadopago.')
```

Las rutas se dividen en dos grupos:

- ruta de webhook: siempre disponible
- demo routes: protegidas por `mercadopago.demo`

## Puerta de demo routes

`EnsureDemoRoutesEnabled` aborta con `404` salvo que:

- `mercadopago.enable_demo_routes` sea `true`
- el entorno de la app sea `local` o `testing`

Eso mantiene las herramientas demo fuera del tráfico de producción, aunque las rutas sigan registradas.

## Flujo de request

Flujo típico de un demo endpoint:

1. un `FormRequest` valida el payload entrante
2. un controller llama a un service del paquete
3. el service usa `MercadoPagoClientFactory`
4. el factory resuelve credenciales y configura el SDK
5. `AbstractApiController` envuelve la respuesta como `{ ok, data, meta }`

## Mapeo de errores

`AbstractApiController` convierte excepciones en respuestas HTTP:

| Tipo de excepción | Status |
| --- | --- |
| `MPApiException` | status code del SDK |
| `MercadoPagoConfigurationException` | `422` |
| `InvalidWebhookSignatureException` | `401` |
| `HttpExceptionInterface` | status code del framework |
| Cualquier otra excepción | `500` |

## Resolución de runtime environment

`MercadoPagoClientFactory` define el runtime del SDK a partir de:

- `MERCADOPAGO_RUNTIME_ENVIRONMENT` cuando existe
- si no, `local` para `local/testing`
- si no, `server`

## Camino del webhook

`WebhookService` procesa requests de webhook así:

- lee el payload
- resuelve `webhookSecret`
- valida `x-signature` cuando hay un secret configurado
- devuelve `acknowledged`, `validated`, `topic`, `resource` y `payload`
