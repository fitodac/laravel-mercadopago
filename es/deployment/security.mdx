## Prioridades de seguridad

Para este paquete, los principales controles son:

- proteger `MERCADOPAGO_ACCESS_TOKEN`
- validar webhooks con `MERCADOPAGO_WEBHOOK_SECRET`
- mantener las demo routes fuera del tráfico de producción
- aplicar tu propio auth y autorización a los endpoints de negocio

## Validación de firma del webhook

Cuando `MERCADOPAGO_WEBHOOK_SECRET` está configurado y existe `x-signature`, `WebhookService` valida la request.

Entradas de validación:

- `x-signature`
- `x-request-id`
- `data.id` desde query string o payload

Formato del manifiesto:

```text
id:{resourceId};request-id:{requestId};ts:{timestamp};
```

El paquete calcula un HMAC-SHA256 y lo compara con `v1` del header.

Si la validación falla, el controller devuelve `401`.

## HTTPS

Usá HTTPS para cualquier webhook endpoint en producción.

Si tu app necesita forzar URLs HTTPS, podés aplicar:

```php
public function boot(): void
{
    if ($this->app->environment('production')) {
        \Illuminate\Support\Facades\URL::forceScheme('https');
    }
}
```

## Manejo de credenciales

- guardá credenciales en `.env` o en un secret manager
- rotá credenciales cambiando el entorno, no el código
- nunca expongas `accessToken` al frontend
- exponé `publicKey` solo cuando los flujos frontend de Mercado Pago lo necesiten

## Exposición de endpoints demo

Nunca trates los endpoints demo del paquete como una business API pública.

Intencionalmente no agregan:

- auth
- autorización
- throttling
- validación de dominio
- persistencia

## Patrón recomendado en producción

Usá tu propio controller e inyectá services del paquete:

```php
Route::middleware(['auth:sanctum', 'throttle:10,1'])->group(function () {
    Route::post('/checkout/preferences', CheckoutPreferenceController::class);
});
```
